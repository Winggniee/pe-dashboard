# 🚦 团队人力分布系统 (Traffic Light Capacity System)

## 概述

新的人力分布系统使用交通灯🚦的概念，将PE按照可用性分为三类，帮助导师快速判断谁可以接收新项目。

---

## 🔴 深度工作 (Red - Busy)

### 定义
正在进行需要深度专注的核心开发工作，达到一定数量阈值

### 判定条件（满足任一即可）
- **超过2个"调优中"项目** (3个或以上)
- **超过2个"搭建中"项目** (3个或以上)
- **超过10个"维护中"项目** (11个或以上)
- **调优中+搭建中+测试中 总数 > 2** (3个或以上)

### 包含状态
- **调优中** - 性能优化阶段（>2个，即3+）
- **搭建中** - 项目初期搭建（>2个，即3+）
- **测试中** - 测试验证阶段（与其他高强度项目合计>2个）
- **维护中** - 日常维护（>10个，即11+）

### 可用性
❌ **不建议分配新项目**
- 这些PE正处于高强度工作状态
- 项目数量已达到饱和
- 新项目会严重影响现有工作质量

### 识别方式
- PE满足上述任一条件
- 图表上显示 🔴 标记
- 卡片背景为红色渐变

### 示例
```
张三的项目状态：
- 调优中: 3个 ✅ (>2，触发深度工作)
- 维护中: 5个
→ 分类: 🔴 深度工作

李四的项目状态：
- 搭建中: 2个 ❌ (=2，不触发)
- 测试中: 1个 ❌ (总计3，但需>2)
- 维护中: 3个
→ 分类: 🟢 空闲 (因为搭建+测试=3，刚好等于不算)

王五的项目状态：
- 搭建中: 1个
- 测试中: 2个 ✅ (总计3个，>2，触发深度工作)
→ 分类: 🔴 深度工作

赵六的项目状态：
- 维护中: 11个 ✅ (>10，触发深度工作)
→ 分类: 🔴 深度工作

钱七的项目状态：
- 调优中: 2个 ❌ (=2，刚好不算深度工作)
- 维护中: 3个
→ 分类: 🟢 空闲 (可以接新项目！)
```

## 🟡 半阻塞 (Yellow - Semi-Blocked)

### 定义
有项目在等待客户反馈，但可以接受小任务

### 包含状态
- **等待中（等客户）** - 等待客户响应，有零散时间

### 可用性
⚠️ **可以分配轻量级任务**
- 可以接收简单项目或维护任务
- 适合分配需要快速响应但不占用大块时间的工作
- 避免分配高难度或时间紧迫的项目

### 识别方式
- PE没有"搭建中"/"测试中"/"调优中"项目
- 但有"等待中（等客户）"项目
- 图表上显示 🟡 标记
- 卡片背景为黄色渐变

---

## 🟢 空闲/商务阻塞 (Green - Available)

### 定义
完全可以接收新项目的PE

### 包含状态
- **等待中（等商务）** - 等待商务部门，PE完全空闲
- **维护中** - 仅有日常维护工作，负载很轻
- **未知状态** - 状态不明确，可能有容量

### 可用性
✅ **优先分配新项目**
- 这些PE有充足时间和精力
- 可以接受任何难度的新项目
- 是项目分配的首选对象

### 识别方式
- PE只有"等待中（等商务）"/"维护中"/"未知状态"项目
- 没有深度工作项目，也没有等客户的项目
- 图表上显示 🟢 标记
- 卡片背景为绿色渐变

## 使用方法

### 1. 查看整体分布

在图表顶部，三张卡片实时显示：
```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 🔴 深度工作   │  │ 🟡 半阻塞     │  │ 🟢 空闲/商务 │
│ 搭建/测试/调优│  │ 等客户        │  │ 等商务        │
│   5 人       │  │   3 人        │  │   2 人        │
└──────────────┘  └──────────────┘  └──────────────┘
```

### 2. 点击卡片查看PE名单

**点击任意卡片**，下方会展开显示该类别的所有PE：

**示例 - 点击🟢空闲/商务阻塞：**
```
🟢 可分配的PE：
┌────────┐ ┌────────┐
│ 张三   │ │ 李四   │
└────────┘ └────────┘
```

### 3. 再次点击收起

点击已选中的卡片，会收起PE名单

## 实际应用场景

### 场景1: 分配紧急高难度项目

1. **查看人力分布卡片**
   - 🔴 深度工作: 6人 (忙碌，不要打扰)
   - 🟡 半阻塞: 2人 (可以考虑)
   - 🟢 空闲/商务阻塞: 2人 ✅ (首选！)

2. **点击🟢卡片**
   - 查看可用PE名单
   - 选择技术能力匹配的PE

3. **分配决策**
   - 优先分配给🟢空闲PE
   - 避开🔴深度工作PE

### 场景2: 分配简单维护任务

1. **查看人力分布**
   - 🟢 空闲: 0人 (没人完全空闲)
   - 🟡 半阻塞: 4人 ✅ (可以接简单任务)
   - 🔴 深度工作: 6人 (太忙)

2. **点击🟡卡片**
   - 查看半阻塞PE
   - 这些PE在等客户，有零散时间

3. **分配决策**
   - 分配简单快速的任务
   - 避免分配需要长时间专注的工作

### 场景3: 团队负载平衡检查

1. **查看整体分布**
   - 🔴 深度工作: 8人 (80%) ⚠️ 负载过重！
   - 🟡 半阻塞: 2人 (20%)
   - 🟢 空闲: 0人 (0%)

2. **采取行动**
   - 团队整体负载过重
   - 暂停接新项目
   - 或考虑调整资源分配

## 技术实现

### 分类逻辑

```javascript
const getAvailabilityCategory = (pe) => {
  const breakdown = pe.statusBreakdown || {};
  
  // Count high-intensity work projects
  const optimizeCount = breakdown['调优中'] || 0;
  const buildCount = breakdown['搭建中'] || 0;
  const testCount = breakdown['测试中'] || 0;
  const maintainCount = breakdown['维护中'] || 0;
  
  // 🔴 深度工作 - Criteria:
  // - >2 projects in 调优中 OR
  // - >2 projects in 搭建中 OR
  // - >10 projects in 维护中 OR
  // - 调优中+搭建中+测试中 total > 2
  const highIntensityCount = optimizeCount + buildCount + testCount;
  
  if (optimizeCount > 2 || buildCount > 2 || 
      maintainCount > 10 || highIntensityCount > 2) {
    return 'busy';
  }
  
  // 🟡 半阻塞 - 有等客户项目
  if ((breakdown['等待中（等客户）'] || 0) > 0) {
    return 'semi-blocked';
  }
  
  // 🟢 空闲 - 其他所有情况
  return 'available';
};
```

### 优先级规则

1. **深度工作判定（任一满足）:**
   - 调优中项目 > 2个 (即 3个或以上)
   - 搭建中项目 > 2个 (即 3个或以上)
   - 维护中项目 > 10个 (即 11个或以上)
   - 调优中+搭建中+测试中 总数 > 2 (即 3个或以上)
   
2. **半阻塞判定:**
   - 不满足深度工作条件
   - 有"等待中（等客户）"项目
   
3. **空闲判定:**
   - 其他所有情况

## 建议的工作流程

### 每日晨会 (Daily Standup)
1. 打开dashboard
2. 查看人力分布卡片
3. 了解团队整体可用性
4. 讨论是否可以接新项目

### 项目分配 (Project Assignment)
1. 有新项目时，先看人力分布
2. 点击🟢卡片，查看可用PE
3. 优先分配给空闲PE
4. 记录分配决策

### 负载平衡 (Load Balancing)
1. 定期检查人力分布
2. 如果🔴深度工作超过70%，考虑：
   - 暂缓接新项目
   - 调整现有项目优先级
   - 增加团队资源
